For several years now I’ve been working on a side project in the realm of distributed data. I’ve given a few presentations about it in the past but they failed to generate much interest and I think this was largely due to an over emphasis on the low-level details at the expense of answering the much more important question of “what exactly is this thing and why should I care?”. It seems like a simple question but when I attempted to write out an answer I quickly discovered why I had unconsciously avoided doing so earlier. There isn’t a simple explanation. Most distributed data systems are straight-forward to describe by way of an “is a” statement like “X is a Y”. For example, “the dsNet is an erasure-coded, object storage system”, “Cassandra is a distributed, eventually-consistent noSQL database”, “Kafka is a distributed message queue and stream processing system”. For Aspen though, there is no “Y”. It doesn’t fall into any of the established categories in distributed data systems so I can’t describe it in simple terms. I do have an “is a” statement for Aspen but its meaning probably won’t be immediately apparent: Aspen is a flexible, low-level building block that allows distributed systems and applications to be designed and implemented in terms of distributed data.

The best way to describe designing and implementing something in terms of distributed data is probably to look at the dsNets’s distributed index. It’s an index designed and implemented in terms of distributed data (objects on the dsNet). Aspen greatly expands upon that simple concept by applying it to entire systems and providing the tools needed to optimize those.systems to their operational environments. Instead of using the conventional approach where the details of distributed data management are abstracted away behind opaque APIs, Aspen instead provides very low-level access to the underlying data model and allows every aspect of it to be leveraged in the design of higher-level applications. How the data is replicated/erasure coded, where it’s located geographically, what kinds of backing media and storage engines are used, approaches to data migration, desired latency and reliability characteristics, and so on are all tradeoffs that can be tuned to best address whatever problem domain is being targeted.

That’s about as accurate and concise a definition as I can provide about what Aspen is but it does little to address the question of why it’s worth pursuing since how it might be used to address real-world problems is probably not immediately obvious. There isn’t much prior art to point I can point to for this so the only approach I can think of to demonstrate how Aspen can be used to so design and implement solutions to real-world problems is by doing exactly that; designing and implementing a solution to a real world problem and then walking through how it works and what the tradeoffs are.



Aspen Overview

The purpose of Aspen is to serve as a flexible, low-level building block for designing distributed systems and applications. Unlike the conventional systems that abstract away the details of distributed data management behind opaque APIs, Aspen instead provides very low-level access to the underlying data model and allows every aspect of it to be leveraged in the design of higher-level applications.  How the data is replicated/erasure coded, where it’s located geographically, what kinds of backing media and storage engines are used, approaches to data migration, desired latency and reliability characteristics, and so on are all tradeoffs that can be tuned as part of the overall system design. Another way to put it is that whereas most conventional distributed systems use distributed data, Aspen-based applications are designed and implemented in terms of distributed data.

Core Concepts:

Data Stores - As the name implies, Data Stores store object slices/replicas. They are relatively small, usually significantly less than the size of a single disk, and they may be freely migrated between backing devices and physical storage nodes (keeping them relatively small helps with migration). 

Storage Pools - Storage pools contain Data Stores that have a data dependency relationship (they host slices/replicas of the same objects). Each Data Store belongs to a single pool.

Objects - Objects are discrete units of data allocated from storage pools and come in two flavors Data and KeyValue. Data objects store opaque, binary data and support overwrite and append operations. KeyValue objects on the other hand contain a mapping of keys to tuples of (revision, HLC timestamp, value) and support concurrent, non-conflicting transactions that operate on different KV pairs within the same object. Objects store the revision and HLC timestamp of the last transaction that successfully updated the object as well as a reference count. In place of a specific delete operation, Aspen objects are freed when their reference count is set to zero. As compared to the dsNet, objects in Aspen more closely resemble dsNet sources than objects as they should generally be kept small (tens of megabytes or less). 

ObjectPointer - Unlike the dsNet, Aspen does not use a namespace. Instead all objects are located by a pointer (~50 bytes in size) that includes a small amount of embedded metadata describing which data stores the object’s slices/replicas are stored on. 

Transaction - All object modifications are made via immediately consistent transactions. Similar to CASN, transactions may simultaneously update multiple objects. Unlike CASN though, they support a much larger array of options for describing the required conditions that must be met for a transaction to commit. For example the following commit criteria could be used: “DataObject A must be exactly revision 1 and with a refcount of 2. KeyValueObject B must not contain the key “foo”, key “bar” must have a timestamp greater than 5 and less than 100, and the revision of the KeyValue object itself must be 1. The local clock on the store must be less than 200”

Finalization Action - Finalization actions are attached to transactions and are used to guarantee the execution of follow-on activities after a transaction successfully commits. They are similar to intents in the dsNet but use a stronger durability model where stores are not allowed to forget the transaction results for successful transactions until all finalization actions have completed. These provide an “at least once” guarantee and should be implemented in an idempotent manner as transaction recovery mechanisms could result in multiple executions.

Derived Concepts:

Tiered Key Value Lists (TKVL) - These are Aspen’s analogue to the dsNet’s distributed index and they’re structured and used in a similar manner. They consist of a bottom linked list that containing the data and a layer of upper tiers that point to the lower tiers to form a B-tree-like structure that provides fast lookup and insert operations for large quantities of data. The primary difference between Aspen’s TKVL structure and the dsNet’s distributed index is in contention mitigation and write-amplification avoidance. Efficiency of the TKVL structure relies upon the fact that transactions acting on different keys within the same object do not conflict and that only the desired operations to apply to the object content are sent in the transaction. Rather than overwriting the entire object every time a key-value pair is added or removed, transactions instead send “Add key foo with value bar” or “Delete key baz”.

Similar to the dsNet’s index, node splits and joins result in pointers to the new/removed node being updated in the parent tier of the tree. The TKVL uses Finalization Actions attached to the splitting/joining transaction to ensure that the parent tier is updated accordingly shortly after the split/join transaction commits. This avoids the increased potential for contention and therefore transaction failure on data insert/remove operations at the cost of the TKVL tier structure bing eventually consistent. Similar to the dsNet’s index though, navigation within a single tier is completely consistent to the same strategies may be used for dealing with missing/invalid pointers when navigating the TKVL structure.

Object Allocaters - Object Allocaters provide a simple and abstract interface for allocating objects and they serve two primary purposes. The first is that they prevent application-level code from needing to directly deal with the complexities of managing a potentially vast numbers of storage pools. The second is that they allow for deployment-time customization of how storage pools should be broken out and organized. Allocaters can also be customized to suit application-specific needs. For example, when objects are allocated for TKVL nodes, the allocated is provided with the tier the node is being allocated for. During deployment-time configuration, allocaters for certain TKVL trees could be configured to allocate upper their nodes as replicated objects backed by SSD media to optimize lookup speed while the data itself is stored as erasure-coded data backed by traditional hard drives. 

DurableTask - Finalization Actions are good for simple, short duration post-transaction commit activities like adding or removing content from a TKVL structure but they have a few downsides. The primary ones being that they require stores to retain the transaction state while they remain outstanding and they can be executed multiple times if the transaction recovery mechanism kicks in. For longer-running operations, particularly those involving multi-step operations or those that required exactly once semantics, Durable Tasks will generally be a better alternative. The idea behind a DurableTask is that an Aspen object is used to store the state associated with the long-running, multi-step task and every time an action is taken that should use exactly-once semantics, the transaction preforming that operation should also include an update to the DurableTask state object indicating that the action was taken. This ensures that crash recovery and concurrent attempts to execute the same task cannot successfully preform the same operation twice.
 
DurableService - Durable Services are similar to Durable Tasks but they are intended to represent long-lived, service oriented tasks that should be running at all times. Like Durable Tasks, each service has an associated state node and they update it regularly to indicate to peers that they are still alive. Should a peer see that the last-modified timestamp exceeds a configured threshold, it may attempt to take over execution of the service by instantiating the service locally and having it successfully update the state object. To avoid potential consistency issues arising from split-brain scenarios, services may use a variant of Master Leases where successfully updating the service state object grants that instance of the service authority to commit transactions for a limited time after the update completed. All transactions executed on behalf of the service may then add a transaction commit requirement that specifies data stores may only vote to commit the transaction if their local clocks are less than the lease expiration time.
